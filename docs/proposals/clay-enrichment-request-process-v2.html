Clay Enrichment Request Process

Standard Operating Procedure for Data Enrichment Requests
12 Labs / Sean Graham March 2026
SOP v2 — Updated with Parent-Child Routing

1. Purpose

This document defines the standard process for submitting, scoping, executing, and delivering data enrichment requests through Clay. It ensures every request is:

  • Properly scoped with clear deliverables
  • Enriched using the right data sources and workflows
  • Delivered back to the CRM in a clean, actionable format
  • Correctly routed to the right company — including parent-child structures New

2. Accepted Input Formats

Enrichment requests can be submitted in any of the following formats:

  • CSV / Excel file — uploaded directly to the shared drive or sent via Slack
  • HubSpot list — a saved list of contacts or companies in HubSpot (provide the list name or URL)
  • Clay table — an existing Clay table that needs additional enrichment columns
  • Manual request — a Slack message or email describing the target audience (e.g., "All VP+ contacts at companies with 500+ employees in fintech")
Preferred format: HubSpot list or CSV with at least name + company + email or LinkedIn URL. The more structured the input, the faster the turnaround.

3. Required Fields

Contact Enrichment

Required

  • Full Name (first + last)
  • Company Name

Strongly Recommended

  • Work Email
  • LinkedIn Profile URL
  • LinkedIn Company URL
  • Job Title
Parent-Child Note: LinkedIn Company URL becomes critical for routing when contacts belong to parent-child structures. Without it, contacts who use a parent company email domain (e.g., @nfl.com) cannot be reliably routed to the correct child company (e.g., Kansas City Chiefs). Always include LinkedIn Company URL when enriching contacts under known parent-child relationships. New

Company Enrichment

Required

  • Company Name

Strongly Recommended

  • Company Domain
  • LinkedIn Company URL
  • HubSpot Company ID (if updating existing records)
Parent-Child Note: LinkedIn Company URL is the #1 signal for correct child company association. When enriching companies that may be part of a parent-child structure, LinkedIn URL determines whether the company maps to an existing child in our mapping table or needs to be flagged as a new child entity. This field should be treated as required for any request involving known parent companies. New

4. Enrichment Scope

Every enrichment request must specify which actions are needed. Check all that apply:

Enrichment Action Yes / No Notes
Contact data enrichment (email, phone, title, LinkedIn) Standard Clay enrichment waterfall
Company data enrichment (revenue, headcount, industry, tech stack)
Lead scoring / ICP fit Requires ICP definition on file
CRM push (create/update contacts or companies in HubSpot) Specify create vs. update
List segmentation Specify segmentation criteria
Parent-child company routing Must be explicit. If yes: contacts routed using LinkedIn-first logic. If no: standard email-domain association. New
When to check "Yes" for parent-child routing: Any enrichment request where target contacts may belong to subsidiaries, divisions, or child companies under a known parent. When in doubt, ask: "Does this parent company have child entities in HubSpot?" If yes, parent-child routing is required. New

5. CRM Handling & Routing

Contact Creation Rules

  • Duplicate check: Before creating a new contact in HubSpot, Clay checks by email address. If a match exists, the record is updated — not duplicated.
  • Owner assignment: Contacts are assigned based on the company's existing owner. If no owner exists on the company, the contact goes to the default rep for that segment.
  • Lifecycle stage: New contacts are created as "Lead" unless the request specifies otherwise.

Deal & Company Rules

  • Existing deals: If a contact is associated with an open deal, enrichment updates the contact but does not change deal ownership or stage.
  • Company association: Contacts are associated to companies by email domain match, unless parent-child routing overrides this (see below).
  • No auto-create companies: Unless the request explicitly asks for company creation, Clay only updates existing company records.

Parent-Child Routing New

When an enrichment request includes parent-child routing (Section 4), the standard email-domain association is replaced with a multi-signal routing approach.

Routing Priority

Priority Signal Why
1st LinkedIn Company URL Exact match against the parent-child mapping table. Most reliable — no fuzzy matching needed.
2nd LinkedIn Company Name If the URL doesn't match a known child, check the company name. Catches subsidiaries not yet in the mapping table.
Fallback Work Email Domain Only used when LinkedIn data is missing or inconclusive. This is the old default behavior — now intentionally last.

Ownership Conflict Rule

When two reps own contacts that route to the same child company, the enrichment request must specify who gets ownership. Options:

  • First-in wins: The rep who already owns the most contacts on the child keeps ownership
  • Submitter decides: The person submitting the enrichment request designates the owner
  • Flag for review: Ownership conflicts are flagged and resolved manually before processing

Personal Email Handling

Rule: Contacts with personal email addresses (@gmail.com, @yahoo.com, @outlook.com, etc.) are never auto-routed to a child company. They are flagged for manual review, regardless of LinkedIn data.

Edge Cases

  • Title says child, LinkedIn says parent: If a contact's job title mentions a child company (e.g., "Director, Kansas City Chiefs") but their LinkedIn Company URL points to the parent (NFL) — the contact is flagged for rep review, not auto-routed.
  • No LinkedIn data + parent email domain: Contact stays on the parent. Cannot differentiate without additional signals.
  • LinkedIn URL matches child but email matches a different child: LinkedIn URL wins. Flagged as a potential data quality issue.

6. Lead Creation Rules

  • Naming convention: Leads are named using the format: [Client Name] [Campaign Name] [Year] (e.g., "NAB Outreach 2026")
  • Source tracking: All leads created through Clay are tagged with Lead Source = Clay Enrichment
  • Campaign association: If the enrichment request is tied to a specific campaign, leads are associated to that campaign in HubSpot
  • Batch tagging: Each enrichment batch gets a unique batch ID for tracking and rollback
  • Child company leads: If a child company is created during enrichment, the lead is created under the child — not the parent. Lead naming includes the child company for clarity. New
    • Example: NAB Outreach 2026 — Kansas City Chiefs

7. Default Rules

These defaults apply unless the enrichment request explicitly overrides them:

  • Duplicates: Update existing records, never create duplicates
  • Missing data: Leave blank fields empty — do not populate with placeholder values
  • Phone numbers: Only enrich with direct dials, not company switchboard numbers
  • Email validation: All emails are verified before CRM push — invalid/catch-all emails are flagged, not pushed
  • Data freshness: Enrichment data must be <90 days old. Older data triggers a re-enrichment
  • No outreach without review: Enriched data is delivered for review before any automated outreach is triggered
  • Parent-child routing (LinkedIn-first): Contacts under parent-child structures are routed via LinkedIn Company URL first, with email domain as fallback New
  • Low-confidence matches stay on parent: When routing confidence is low (no LinkedIn data, ambiguous signals), contacts remain on the parent company — never auto-routed to a child on weak evidence New
  • Personal emails flagged: Contacts with personal email addresses (@gmail, @yahoo, etc.) under parent-child structures are always flagged for manual review — never auto-routed New
  • Existing associations respected: Existing parent-child company associations in HubSpot are respected unless the enrichment request explicitly requests re-evaluation New

8. Delivery

Every enrichment request is delivered with the following:

Standard Delivery

  • Enriched Clay table — the completed table with all enrichment columns populated
  • Summary report — total records processed, match rates per data point, records skipped (with reasons)
  • CRM push confirmation — list of records created/updated in HubSpot with record IDs
  • Flagged records — any records that need manual review (bad data, duplicates, missing required fields)

Parent-Child Routing Segment New

For requests with parent-child routing enabled, delivery includes an additional routing segment:

  • Auto-routed: Contacts automatically associated to the correct child company (count + list)
  • Kept on parent: Contacts that stayed on the parent — all signals agreed they belong there (count + list)
  • Flagged for review: Contacts that need rep attention — ownership conflicts, personal emails, ambiguous signals (count + list)

Routing Confidence per Record

Confidence Meaning Action Taken
High LinkedIn URL matched a known child in the mapping table Auto-routed to child
Medium LinkedIn company name matched, or email domain matched a child domain Auto-routed with flag for spot-check
Low Signals conflicted, or only job title suggested a child Kept on parent, flagged for rep review

Review Process

  • Spot-check: A random sample (5-10%) of enriched records is manually reviewed before delivery
  • Requester sign-off: The person who submitted the request reviews the summary report and flags any issues before CRM push
  • Rollback window: All CRM pushes can be rolled back within 48 hours using the batch ID

10. Parent-Child Routing Reference New

Consolidated reference for all parent-child routing logic. Use this section when submitting or processing enrichment requests that involve parent-child company structures.

Routing Decision Flowchart

New contact enters enrichment | |-- Has LinkedIn Company URL? | |-- Matches a known child in mapping table? | | YES --> Associate to that child company [HIGH confidence] | | NO --> Check LinkedIn company name... | | | |-- LinkedIn company name matches a known child? | YES --> Associate to that child company [MEDIUM confidence] | NO --> Check work email domain (fallback) | |-- No LinkedIn data? | |-- Has work email? | | |-- Email domain matches a known child domain? | | YES --> Associate to that child [MEDIUM confidence] | | NO --> Keep on parent [LOW confidence] | | | |-- No work email either? | --> Flag for manual review | |-- Personal email (@gmail, @yahoo, etc.)? | --> Flag for rep review (never auto-route) | |-- Confidence too low on any match? --> Flag for rep review (task created in HubSpot)

What Submitters Must Include for Parent-Child Requests

When submitting an enrichment request that involves parent-child company structures, include:

  1. Parent company name(s) — which parent companies are involved in this request
  2. Known child companies — list any child companies you're already aware of (even if incomplete)
  3. LinkedIn Company URLs — for both parent and child companies (critical for mapping table)
  4. Ownership rules — how to handle conflicts when two reps own contacts routing to the same child (see Section 5)
  5. New child handling — should Clay create new child companies in HubSpot if discovered, or flag them for team review first?

How New Child Companies Get Added to the Mapping Table

The parent-child mapping table is the source of truth for routing. New entries are added through:

  1. Migration discovery (Proposal 1): Claude analyzes existing HubSpot data, identifies child companies, and the team approves them via Google Sheet review. Approved children are added to the mapping table.
  2. Enrichment discovery: During enrichment processing, if a contact's LinkedIn Company URL points to a company not in the mapping table but clearly a subsidiary — it's flagged as a potential new child. The team reviews and approves before it's added.
  3. Manual addition: A rep or manager identifies a new child company and requests it be added. Requires the child company's LinkedIn URL and the parent company it belongs under.
  4. Mismatch alert: The weekly re-association audit (automated) surfaces contacts whose company association was manually changed by a rep — indicating the routing may need a new child entry.
Key principle: The mapping table grows over time. Every enrichment request is an opportunity to catch new child companies. The system gets smarter with every batch processed.
